h2. 機能
本質的には、Chef ホスティングは集中管理されたデータストアであり、Chef クライアントたちはこのデータストアへと IP アドレスや読み込まれたカーネルモジュールの一覧、OS バージョン等々のシステム情報を保存します。このようなデータは、いくつかの重要な観点からあなたのインフラ全体にとって非常に便利なものとなります：

!opscode-platform-about.png|align=right,hspace=18,vspace=6!
h3. * {color:#f7681a}*検索ベースの自動化：*{color}
Chef ホスティングによって収集されるすべてのデータはインデクスされ検索可能となっています。このデータに対してレシピの中から動的に問い合わせを行えるので、複雑な設定が必要となるサービスでも設定することができます。たとえばですが：動的に MySQL のマスタサーバを検出する、nagios を設定する、クラスタ内のすべての memcached サーバを見つける等。

h3. * {color:#f7681a}*ロールベースの権限管理：*{color}
このようなデータに対するインデクスの一番上の層には、きめ細かい権限管理システムがあり、このおかげで従業員や受託会社その他のさまざまなグループにについて、どの程度インフラへアクセスを許すか集中管理することができます。監査役にはすべてのインフラについてリードオンリー権限を与えたいとか、DBA を担当する受託会社には MySQL のスレーブサーバに対するアクセス権だけ与えたいとか。

h3. * {color:#f7681a}*クックブックの配布：*{color}
Chef ホスティングなら、インフラを定義するクックブックとロールを集中管理することができます。開発環境の Apache ウェブサーバをすべて更新したいとして、Chef ホスティングがデータセンターだろうがクラウドだろうが、どこに変更を適用すればよいか把握していてくれます。

h3. * {color:#f7681a}*移植性：*{color}
Chef ホスティングに保存されたデータはインフラの設計図として機能するので、本番環境とまったく変わらないクローン環境をいくつでも簡単に作れるようになります。

h2. スケールするように構築されています
Chef ホスティングは Opscode によって頭からつま先まで完全に、たくさんのユーザが使えて水平方向にスケールするように構築されています。{color:#f7681a}Opscode を利用するなら、システム管理ツールが限界に達してインフラやビジネスについての需要を満たせなくなる心配は不要です。

ビジネスにとって構成管理が重要になってくるにつれ、可用性の高いシステムを保守するコストもかなりかさんでいきます。 _もし管理するサーバが 100 台規模から 1000 台規模へと増えるのだとしても、Chef ホスティングが負荷をうまく処理してくれるので、管理コストが一緒に増えてしまうということはありません。_

*新しいバージョンの Chef や Opscode API がリリースされたときでも、Chef ホスティングならすぐに最新版が使えます* ーーデータ移行や後方互換性の面倒も見てくれるので、大きなアップグレードの際でも作業する必要がなくなるというわけです。
